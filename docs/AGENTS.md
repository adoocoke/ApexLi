# AGENTS.md - EA Agent 项目 Agent 协作规范

> 本文档面向所有参与 EA Agent 开发的 Agent（包括 Grok），定义协作规则和最佳实践。

## 1. 核心原则

- **最小有效流程**：优先使用最简单的方式解决问题。只有在反复证明必要时才增加规则。
- **Agent First**：优先依赖模型能力和工具，而不是繁重规则和 Prompt。
- **Test First**：新增行为、修复缺陷、重构公共逻辑时，默认使用 TDD。
- **清晰交接**：任何修改都应考虑下一个接手的人（包括未来的 Agent）。
- **快速反馈**：小步提交、频繁验证、及时修复。

## 2. 工作流程（推荐循环）

推荐使用以下 XP 风格循环：

**Orient → Clarify → Slice → Check → TDD / Implement → Verify → Reflect**

- **Orient**：理解当前上下文和目标
- **Clarify**：当意图、范围、验收标准不清时，必须先提问澄清
- **Slice**：把任务切成可独立验证的小增量
- **Check**：在改动前定义验收信号
- **TDD / Implement**：优先写测试，再实现最小可通过代码
- **Verify**：验证是否满足验收信号
- **Reflect**：总结经验，必要时更新文档或规则

## 3. Plan vs Spec

- **Plan**：适合一次会话内能完成的工作。使用 TODO / checklist 跟踪进度。
- **Spec**：适合跨会话、跨 Agent、或需要长期留存的复杂工作。使用 `docs/specs/` 目录。
- 事故处理优先使用 Plan 模式（先恢复 → 缓解 → 验证 → 事后记录）。

## 4. 文档规范

- **代码和测试** 是最高权威的活文档。
- 新增重要决策使用 **ADR** (架构决策记录)，放在 `docs/decisions/`。
- 长期知识优先迁移到代码、测试、`AGENTS.md` 或 ADR，而不是留大量 Spec。
- 根目录保留简洁 README，详细内容放在 `docs/`。

## 5. 测试策略

- **unit/**：纯逻辑测试，不依赖外部（LLM、Tushare 等）
- **integration/**：使用 Mock 的多组件测试（推荐大部分测试放这里）
- **e2e/**：真实调用外部服务（Tushare、真实 LLM）的测试（数量要少）

新增功能时优先写对应层级的测试。

## 6. 沟通与提问规范

当任务意图、范围、取舍、验收标准不清楚时：
- 先使用**发散提问**探索选项
- 再使用**收敛提问**锁定方案
- 提问时尽量基于已观察到的事实给出选项和推荐理由

## 7. 变更纪律

- 修改核心节点（graph、nodes、strategies）前，必须先写或更新测试。
- 大型重构建议使用 worktree 隔离。
- 重要变更建议先在小范围验证（小步提交 + 快速反馈）。

## 8. 更新本文件

本文件应随项目演进持续更新。当发现更好的实践时，及时补充到本文件中。
