# Martin Fowler Harness 审查报告

**项目**：eaagent  
**审查日期**：2026-06-18  
**审查人**：Grok（基于 Martin Fowler 的 Harness 与重构思想）

---

## 1. 总体评价

当前项目已经具备了 **Agent Harness 的基本框架**，尤其在**可观测性**上做得比较好。但在 **Guides 的强度** 和 **Sensors 的智能程度** 上，还有明显提升空间。

整体成熟度评分：**7.2 / 10**（早期可用阶段）

---

## 2. 核心维度检查

| 维度                  | Martin Fowler 核心要求               | 当前实现评价     | 得分   | 说明 |
|-----------------------|--------------------------------------|------------------|--------|------|
| **Guides（前馈控制）**    | 通过规则、Playbook 主动引导行为       | Playbook 仅被动注入 | 6/10   | Guides 偏弱，是当前最大短板 |
| **Sensors（反馈控制）**   | 实时感知状态、发现问题、提供反馈     | 有双层 Sensors     | 7.5/10 | 混合模式思路正确，但智能度不足 |

## 3. 主要问题与改进建议

### 3.1 Guides 偏弱（最高优先级）

**问题**：
- Playbook 目前只是被加载后塞进 system prompt。
- 缺乏对规则的**结构化解析**和**符合度验证**。

**建议**：
- 将 Playbook 拆解为结构化规则（而非纯文本）。
- 在 `llm_critique` 中增加规则符合度打分。
- 考虑引入“规则违反”反馈机制。

### 3.2 Sensors 智能度不足

**问题**：
- `quality_sensor` 仍以硬编码规则为主。
- `llm_critique` 的 Prompt 设计和输出结构还可以进一步优化。

**建议**：
- 保留轻量级确定性 Sensors。
- 加强 LLM-based Sensors 的 Prompt 工程和结构化输出。
- 增加分析质量评分（0-10 分）。

### 3.3 Steering Loop 缺少智能停止条件

**问题**：
- 当前主要靠“问题数量”或“理由数量”来决定是否提前结束。
- 缺少对“分析质量是否已经足够”的判断。

**建议**：
- 在 `llm_critique` 中增加 `analysis_quality_score`。
- 当分数超过阈值时，即使未到最大轮次也可提前结束。

### 3.4 重构仍可继续深化

**已完成**：
- 拆分出 `data_provider.py`、`playbook_loader.py`、`config.py`

**可继续拆分**：
- `sensors.py`（统一管理 `quality_sensor` 和 `llm_critique`）
- `llm_client.py`（统一管理所有 LLM 调用）
- `state.py`（独立管理 `TAState` 定义）

## 5. 总结

当前 eaagent 的 Harness 已经有了比较清晰的框架和良好的可观测性，但在 **Guides 的约束力** 和 **Sensors 的智能程度** 上仍有较大提升空间。

建议下一阶段重点围绕以下两点进行迭代：

1. **让 Playbook 真正成为强 Guides**（而非仅提示）
2. **让 LLM Critique 承担更多“发现问题”和“质量判断”的职责**

## 6. 后续数据喂给 LLM 的三种方案（2026-06-18 新增）

在当前实现中，我们已经把 CSV 的全部历史数据发给了 LLM。但未来在数据量更大或需要更智能分析时，可以考虑以下三种方向：

### 选项 1（推荐现在做）
**把更多历史数据喂给 LLM（比如最近 15~20 根K线 + 关键指标）**

- 优点：LLM 能看到更完整的趋势、形态和量价结构，分析质量会明显提升。
- 缺点：Token 消耗会增加。
- 建议做法：
  - 发送最近 15~20 根K线（OHLCV）
  - 额外计算并附上常用指标（如 MA5、MA10、ATR、最近高低点等）
  - 让 LLM 基于更丰富的数据进行趋势判断和形态识别

### 选项 2
**保持当前轻量方式，但加强 Prompt 结构**

- 优点：Token 消耗低，实现简单。
- 缺点：LLM 看到的信息有限，分析深度受限。
- 建议做法：
  - 保持只发最近 5~8 根数据
  - 大幅优化 Prompt 结构，要求 LLM 必须按固定格式输出（趋势、关键位、量价、交易建议、Playbook 符合度等）

### 选项 3
**引入工具调用（Tool Use / Function Calling）**

- 优点：最智能，LLM 可以自己决定需要哪些指标和数据，再调用 pandas 进行计算。
- 缺点：实现复杂度最高，需要定义工具 schema。
- 建议做法：
  - 让 LLM 作为 Agent，可以调用工具（如 `calculate_ma`、`find_swing_high_low`、`get_volume_profile` 等）
  - LLM 自己规划分析步骤，再得出结论
  - 这是长期最推荐的方向，符合现代 Agent 开发趋势

**推荐优先级**：选项 1 > 选项 3 > 选项 2

## 7. 下一步改进方向（2026-06-18）

当前版本已实现把完整历史数据发给 LLM 进行分析，效果有明显提升。但仍存在一些可以继续优化的方向：

### 选项 A（推荐）
**改进 `quality_sensor`，让它也更智能一点**

- 可以结合 LLM 的观察结果进行判断
- 或者增加一些基于数据的简单规则判断（例如：趋势是否确认、量价是否配合、是否接近关键位等）
- 目标是让确定性检查和 LLM 分析保持一致性，而不是脱节

### 选项 B
**优化置信度计算方式**

- 当前置信度是硬编码的（`0.65 + iteration * 0.08`）
- 可以让 LLM 在生成信号或审查时，同时给出一个置信度分数（0-100）
- 或者结合多个维度（数据质量、Playbook 符合度、信号一致性等）综合计算

### 选项 C
**先保持现状，继续跑更多测试**

- 使用不同时间段、不同品种的 data 进行测试
- 观察在不同市场环境下的表现（趋势市、震荡市、反转初期等）
- 收集更多案例后再决定下一步重点优化方向

**建议优先级**：A > B > C
