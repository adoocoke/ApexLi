### **阶段化演进计划**

#### **Phase 1: 架构重构与 Harness 落地（优先级最高，建议 3-4 周）**

**目标**：建立清晰的 Agent Harness 骨架，让系统具备可观测和可控的基础。

**关键动作**：

1. **定义清晰的 Harness 组件边界**（在代码层面落地）：
   - **Guides**：Playbook + 风险规则 + 时间框架协同规则（结构化存储）
   - **Sensors**：数据获取层（Tushare 封装 + 未来其他数据源）
   - **Actor**：决策执行者（当前 ReAct Agent 的核心）
   - **Steering**：风控与规则仲裁层（最重要的新组件）
   - **Memory**：分层记忆（短期分析记忆 + 长期交易记忆 + 规则执行历史）

2. **LangGraph 状态图重构建议**：
   - 引入 `AnalysisState`（包含各时间框架独立分析结果 + 冲突标记）
   - 增加 `SteeringNode`（在多时间框架分析后、最终决策前强制执行）
   - 考虑使用 **LangGraph 的子图（Subgraph）** 分别处理 1d / 30m / 3m 分析

3. **可观测性基础设施**：
   - 所有关键节点输出结构化日志（尤其是 Playbook 规则检查结果）
   - 引入 `DecisionTrace` 对象，记录：引用了哪些 Playbook 条款、是否通过、冲突原因

**交付物**：
- 新的状态定义（`state.py`）
- `SteeringNode` 实现
- Harness 组件接口定义

#### **Phase 2: Playbook 规则引擎化（核心攻坚，建议 4-5 周）**

这是目前最需要解决的痛点。

**目标**：让 Playbook 从“被引用”变成“被执行”。

**具体方案**（推荐组合使用）：

1. **Playbook 结构化**：
   - 将现有 Playbook 拆解为 **结构化规则集**（建议用 YAML/JSON + Pydantic 模型）
   - 每条规则包含：`id`、`timeframe`、`condition`、`action`、`priority`、`risk_implication`

2. **引入规则执行机制**（推荐两种方式结合）：
   - **方式A（推荐主方案）**：`RulesEngine` 节点
     - 在 LangGraph 中增加独立节点，负责逐条检查当前分析是否满足规则
     - 输出结构化结果：`passed_rules`、`violated_rules`、`skipped_rules`
   - **方式B**：Tool 化关键规则
     - 把部分可计算的规则做成 Tool（如“趋势确认至少两个高低点”）

3. **强制执行策略**：
   - 在 `SteeringNode` 中设置**硬约束**：若存在高优先级规则未满足，直接进入 `NO_TRADE` 路径
   - 要求 Agent 在最终决策前必须调用 `check_playbook_compliance` 工具/节点

**关键设计原则**：
- Playbook 规则应**先于 LLM 推理**被评估，而不是让 LLM 自己判断是否遵守。

---

#### **Phase 3: 多时间框架协同与冲突仲裁机制（与 Phase 2 并行或紧随其后）**

**目标**：把“坚决不做单”的纪律变成系统强制行为。

**建议实现路径**：

1. **独立分析 + 聚合仲裁**：
   - 每个时间框架走独立分析子图（或并行节点）
   - 聚合节点输出 `TimeframeConsensus` 对象，包含：
     - 各时间框架趋势方向
     - 形态识别结果
     - 信号强度
     - **冲突标记**（明确列出冲突点）

2. **冲突仲裁规则**（放入 Steering）：
   - 日线趋势未确认 → 直接否决
   - 30m 与 3m 形态冲突 → 否决
   - 仅在所有时间框架**方向一致 + 无重大形态冲突**时才允许进入交易决策

3. **状态设计建议**：
   ```python
   class TimeframeAnalysis(BaseModel):
       timeframe: str
       trend: str
       patterns: list[str]
       confidence: float
       playbook_rules_checked: list[str]
   ```

#### **Phase 4: 风险控制内嵌化**

**目标**：让 30% 仓位 + 动态止损成为决策的一部分，而不是事后补充。

**具体措施**：

1. 在 `SteeringNode` 中增加**风险评估子流程**：
   - 基于当前分析结果计算建议止损位（可先规则化，后续接入工具）
   - 强制输出 `position_size`（当前固定 30%，后期可演进为动态）
   - 记录风险收益比（R:R）

2. 增加 `RiskCalculationTool`（即使初期是规则驱动的）：
   - 输入：入场价、止损价、ATR 等
   - 输出：建议仓位、最大亏损金额

3. 决策输出结构化：
   - 最终决策必须包含 `risk_assessment` 字段

---

#### **Phase 5: 工具体系升级**

**当前问题**：工具调用较为松散，缺乏结构化。

**演进方向**：

| 维度           | 当前状态       | 目标状态                     | 优先级 |
|----------------|----------------|------------------------------|--------|
| 工具抽象       | 直接调用       | 分层（Data / Analysis / Risk） | 高     |
| 工具选择       | LLM 自主决定   | 结合 Steering 进行约束       | 高     |
| 形态识别       | 依赖 LLM       | 逐步工具化（可验证）         | 中     |
| 风险计算       | 缺失           | 独立工具                     | 高     |
| 工具可观测性   | 弱             | 记录调用参数 + 返回 + 耗时   | 高     |

**建议**：
- 建立 `ToolRegistry`，对工具进行分类和权限控制
- 在 Steering 中可以**禁用或强制要求**某些工具
- 未来形态识别工具应返回**结构化结果 + 置信度**，而非纯文本

#### **Phase 6: 评估、记忆与持续改进**

1. **Memory 设计**：
   - 短期：当前分析会话的完整轨迹
   - 长期：历史交易 + 规则执行记录（用于后续自我评估）
   - 规则执行历史：记录“某次决策引用了哪些规则、是否通过”

2. **自我评估机制**（Harness 的重要组成部分）：
   - 增加 `PostDecisionReview` 节点（可异步）
   - 定期对历史决策进行规则符合度打分

3. **评估指标建议**（后期建立）：
   - Playbook 规则遵循率
   - 多时间框架冲突拦截率
   - 风险参数输出完整率
   - 工具调用准确率
